这是一个非常具有野心且技术路径正确的决定。通过 Hook（挂钩）机制 统一接管 Claude Code、Gemini CLI 和 Codex CLI 的生命周期，是实现“深度编排”的唯一可行方案。

这种架构不再是简单的“多开终端”，而是构建了一个通用 AI 运行时环境 (Universal AI Runtime)，GUI 是这个运行时的控制台。

以下是 Code Flow V10.0 "Hook-Verse" 架构规划。

一、 核心架构：统一 Hook 适配器模式 (Unified Hook Adapter Pattern)
为了让三个完全不同的 CLI 工具（Claude-NodeJS, Gemini-Python/Go, Codex-Python）协同工作，我们需要一个中间层（Middleware Kernel）。

1. 抽象层：ICliAdapter 接口
所有的 AI 引擎（无论是官方 CLI 还是自定义脚本）都必须被封装成一个 Adapter，并暴露统一的 Hook 插槽：

Hook 插槽触发时机GUI 介入逻辑 (Intervention)hook_before_send消息发送给 API 前[Context 治理] 检查 Token 计数；执行自动总结；注入动态 System Prompt。hook_on_stream接收流式响应时[UI 渲染] 实时更新界面；检测是否出现了特定的“唤醒词”或“工具调用符”。hook_tool_interceptAI 决定调用工具时[多 AI 路由] 拦截 call_coder；挂起当前 AI；启动子 AI 流程。hook_after_exec工具/代码执行完后[状态快照] 触发 Git Commit；序列化当前对话栈；写入 SQLite 用于回滚。hook_restore_state发生回滚请求时[状态注入] 强行重置 CLI 内部的 Messages 数组（需各 CLI 配合实现状态重载接口）。

2. 实现层：三大引擎适配器
Claude Adapter: 深度 Hook claude-code (Node.js)。利用 Node 的 require 拦截或对其 API 客户端进行 Proxy 代理，接管其 messages 维护权。
Gemini Adapter: 封装 Google 的 GenAI SDK。由于 Gemini 上下文窗口极大，Hook 重点在于多模态输入处理（自动将图片/文件转为 Gemini 可读格式）。
Codex Adapter: 封装 OpenAI API 或 Azure OpenAI。Hook 重点在于JSON Mode 强制和输出解析，确保其生成的代码能被准确提取。
二、 功能模块：基于 Hook 的全链路编排
1. 配置系统：级联继承 (Cascading Configuration)
原则： 一切皆可配置，一切配置在 Runtime 均可被 Hook 修改。

配置层级：
Global (系统级): 默认 API Key，默认 Git 行为，默认总结阈值 (e.g., 20k)。
Session (会话级): 定义“模式”（如：重构模式、创意模式）。
Role (角色级): 定义 Main / Sub / Coder 三个位置具体填充哪个 Adapter。
GUI 配置面板详情：
主 AI (Main): 配置为 Claude 3.7。设定 SystemPrompt = "你是架构师"。
子 AI (Coder): 配置为 Gemini 1.5 Pro (利用其长窗口)。设定 Hook_AutoSummary = Disabled (Gemini 不需要频繁总结)。
工具集 (MCP):
Main 挂载: ProjectManager, WebSearch.
Coder 挂载: FileSystem (读写权限), Linter.
API 渠道: 为 Coder 配置独立的 BaseURL (如 DeepSeek 专线) 以节省成本。
2. 上下文治理：Hook 驱动的自动总结 (Auto-Summary)
痛点解决： 以前是定时任务，现在是即时拦截。

逻辑流：
用户发送消息。
触发 hook_before_send。
Adapter 计算 Current_Token + New_Msg_Token。
判断：
如果 > Config.summary_threshold (例如 20k)：
阻断发送。
调用 Summary_Agent (可以是轻量级模型) 对前 80% 历史进行摘要。
构建新的 Messages 数组：[System + Summary + 最近 20% 对话]。
调用 Adapter 的 reload_context(new_messages) 方法。
放行： 继续发送用户消息。
3. 状态回滚：双重快照 (Dual-Layer Snapshot)
逻辑： 每次 AI 产生实质性副作用（写代码）前，必须存档。

Checkpoint 结构：
JSON

{
"step_id": 1024,
"git_hash": "f8d9a2b", // 代码状态
"dialog_states": { // 对话状态
"main_ai": { "history_hash": "...", "adapter_data": "..." },
"coder_ai": { "history_hash": "...", "adapter_data": "..." }
}
}
恢复流程：
用户点击“回滚到此”。
GUI 执行 git reset --hard。
GUI 读取 dialog_states。
并行触发 Main 和 Coder 的 hook_restore_state。
Adapter 内部利用官方 CLI 的 rewind 命令或重新初始化 Session，确保 AI 不记得“被回滚掉的错误尝试”。
三、 协作模式：主从 AI 的 Hook 交互
场景： Claude (Main) 指挥 Gemini (Coder) 写代码。

Main AI 思考： 输出 <tool>call_coder(task="Fix bug")</tool>。
Hook 拦截： hook_tool_intercept 捕获该输出，阻止其直接显示在 UI 上。
UI 状态变更： 主对话框显示 “Claude 正在指挥 Gemini...” (折叠状态)。
上下文传递 (Context Grafting):
GUI 提取 Main AI 的 task 参数。
GUI 通过 Code Flow 的 CodexLens 搜索相关代码片段。
组装 Prompt: [Task] + [Related Code] + [Project Rules]。
Coder AI 启动：
初始化 Gemini Adapter。
注入组装好的 Prompt。
Gemini 执行代码修改 (触发 hook_after_exec 提交 Git)。
结果回传： Gemini 完成任务，Hook 将其输出总结为：“已修复 bug，修改了文件 A 和 B”。
Main AI 恢复： 将总结作为 Tool Result 返回给 Claude。
四、 开发者实现路径 (Dev Roadmap)
Phase 1: Adapter 封装 (The Wrappers)
不直接运行 CLI，而是编写三个 Python/Node 类：

ClaudeClient (对接 Claude Code)
GeminiClient (对接 Google GenAI SDK)
CodexClient (对接 OpenAI SDK)
目标： 统一实现 send(), receive(), get_history(), set_history() 方法。
Phase 2: Hook 总线 (The Bus)
实现一个 Event Emitter，GUI 所有的操作都只发送 Event，不直接调用 Client。

Event: USER_SEND_MSG -> 触发 Pipeline -> PRE_HOOK -> CLIENT_SEND -> POST_HOOK.
Phase 3: GUI 配置面板
实现一个动态表单生成器，根据当前选中的 Adapter，渲染其特有的配置项（例如 Gemini 特有的 SafetySettings，Claude 特有的 CacheControl）。
Phase 4: 数据库与回滚
SQLite 表设计：checkpoints 表，存储大 JSON blob。
Git 联动逻辑。
总结
这个架构的核心在于**“去中心化但统一管控”**。

去中心化： 每个 AI 模型（Main/Coder）都有自己独立的 Adapter 和 Context，互不干扰（避免了 Token 浪费）。
统一管控： 所有 Adapter 都受制于 GUI 的 Hook 系统，必须遵守统一的 Token 限制、Git 提交规则和回滚协议。
