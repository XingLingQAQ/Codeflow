D. [Key Evidence & Minimal Reasoning] (关键证据与推论)
1. Key Evidence
AST 优化： 仅传递函数签名（Function Signature）而非全量代码，可降低 60%-80% 的上下文 Token 消耗 [1.1]。
混合检索有效性： FTS 负责解决“确定性名词”查找，Vector 负责“意图”查找，Graph 负责“逻辑链路”查找，三者结合能显著提高召回精度 [2.3]。
记忆分层理论： 模仿人类记忆，将频繁变动的“活动记忆”存入高性能 SQLite，将稳定的“冷记忆”提炼为 Markdown 规则文件（.rules）更有利于模型遵循 [3.5]。
2. Minimal Reasoning
关于拖拽归档： 自动总结虽然方便，但容易产生“信息熵增”。通过 GUI 拖拽，将“机器聚类”的记忆片段交由用户手动确认归档，能极大提高 .claude/rules 的质量。
关于 AST 树： 在发送 Prompt 前，用户对“上下文”的可见可控是解决 AI 幻觉的第一步。
E. [Final Deliverable] Code Flow 核心组件 UI 功能规格
1. 记忆管理仪表盘 (Memory Management Dashboard)
逻辑： 实现从“海量日志”到“结构化规则”的提炼过程。

视觉布局：
左侧：短期记忆池 (Active Sessions)。以“记忆簇 (Clusters)”展示，自动通过语义相似度聚类（如：关于“Redis 优化”的 5 次对话被聚为一簇）。
右侧：长期规则库 (Project Rules)。展示 .claude/rules/ 文件夹下的 Markdown 文件树。
交互逻辑：
拖拽归档 (Drag-and-Drop)： 用户选中一个记忆簇，拖入右侧特定规则文件。
自动提炼 Hook： 触发一个总结 Agent，将该簇内的对话精华（决策、修复方案）提炼为简短的 Markdown 列表，并追加到目标规则文件中。
热度感知： 频繁被引用的记忆簇高亮显示，提示用户该部分需转化为规则。
2. 动态上下文构建器 (Context Builder with AST)
逻辑： 解决“上下文爆炸”和“Token 浪费”的最后一道防线。

AST 树可视化：
文件列表中每个文件可展开为树状结构：File -> Class -> Method/Function。
粒度化选择：
三态复选框： 1. [ ] 不包含。
2. [/] 仅签名 (Signatures Only)：仅包含函数定义、参数及 Docstring。
3. [X] 全量代码 (Full Body)：包含完整逻辑。
Token 实时预估： 底部显示“当前上下文：1,240 tokens / 32,000 max”，随着勾选实时波动。
智能自动勾选： 提供“建议上下文”按钮，根据当前问题自动预勾选最相关的函数签名。
3. 语义检索引擎 (Command-K Search)
逻辑： 一个入口，搜索万物。

全局入口： 快捷键 Cmd/Ctrl + K 呼出中心浮窗（类似 Raycast/Spotlight）。
分类结果展示：
🔴 精准匹配 (FTS)： 快速查找文本中的变量名、硬编码字符串。
🔵 语义相关 (Vector)： 处理“登录逻辑”、“支付流程”等模糊描述。
🟢 引用关系 (Graph)： 显示“调用了此函数的代码”或“被此函数引用的配置”。
一键注入 (Actions)： 搜索结果右侧提供 [Add to Context] 或 [Go to Definition] 按钮。
F. [Uncertainty & Risks] (风险与不确定性)
性能瓶颈 [Uncertain]： 在超大型项目（数万文件）中，实时构建 AST 树并计算 Token 可能导致 GUI 卡顿。
对策： 仅对当前打开的文件或搜索命中的文件进行深度解析，其余使用文件树缓存。
聚类精度 [Evidence insufficient]： SQLite 记忆簇的自动聚类可能出现“分类错误”。
对策： 允许用户在仪表盘内手动解散或重组记忆簇。
AST 兼容性： 不同语言（Python, TS, C++）需要不同的解析器，前端渲染需具备通用性。
