Code Flow V12.0 智能干预与动态编排功能规划
1. 渐进式记忆披露 Hook (Progressive Memory Disclosure)
核心逻辑： 解决“全量加载记忆浪费 Token”与“不加载记忆导致 AI 幻觉”的矛盾。通过 Hook 在发送前进行“语义预检”。

ID功能模块详细逻辑说明MEM-H01语义预检触发器 (Pre-flight Check)Hook 注入点： hook_on_user_input_submitted。

当用户点击发送后，系统先不调用主 AI，而是提取当前 Input 的关键词与向量库/知识图谱进行碰撞。

MEM-H02意图匹配与分级1. 强匹配： 命中 .claude/rules 中的特定文件名（如 DB_Refactor.md）。

2. 弱匹配： 命中 SQLite 中的历史会话簇（如半个月前关于 Redis 的讨论）。

MEM-H03UI 交互：建议性弹窗交互行为： 在输入框上方或屏幕中央弹出半透明浮窗。

• 文案示例： “检测到 2 条与‘数据库重构’相关的长期规则及 1 个历史会话簇。是否载入上下文？”

• 操作： [查看内容] [确定加载] [忽略并发送]。

MEM-H04动态注入逻辑若用户选择“加载”，Hook 会将选中的记忆片段实时注入到当前请求的 Context 头部，并标记为 [Historical Context]。

2. 任务级模型热切换 (Task-Level Model Hot-Swap)
核心逻辑： 允许在对话中途“换马”，应对不同任务对模型能力（或成本）的动态需求。

ID功能模块详细逻辑说明SWAP-01单步覆盖配置 (Turn-based Override)在每一个对话气泡或任务卡片的右上角，提供一个“模型切换”下拉菜单。该操作仅影响当前这一步及基于此步骤生成的子任务。SWAP-02重试与接力逻辑场景 A (重试)： 步骤 1 使用 Claude 失败，用户点击 Hot-Swap 为 o3 并点击“重试”，系统自动执行 /rewind 并用新模型重新请求。

场景 B (接力)： 步骤 1 由 Claude 规划完成，用户发现步骤 2 是纯代码实现，手动切换为 DeepSeek 执行，以节省成本。

SWAP-03PAPI 变量联动用户可以在 Hot-Swap 菜单中直接选择 PAPI 预设变量（如 ${BACKEND_EXPERT}），系统会自动加载该变量对应的模型、API 渠道和特定 System Prompt。SWAP-04上下文平滑迁移切换模型时，Hook 自动执行 on_before_context_fill，根据新模型的 Window Size 自动调整总结方案（例如从 Claude 切换到 Haiku 时，自动加大压缩比）。

3. 深度联动：当 Hot-Swap 遇到 Git 回滚
核心需求： 确保手动切换模型后的状态可追溯。

原子快照更新： 当用户执行 Hot-Swap 并重试时，系统会生成一个新的 Snapshot，记录：“Interaction #42: Model changed from Claude to Gemini; Action: Retry.”
回滚一致性： 如果用户回滚到热切换之前的步骤，所有 Agent（主/子/Coder）必须同步执行回滚，并恢复到该步骤原始定义的默认模型，清除“热切换”的临时配置。
4. GUI 界面增强设计
记忆预警指示灯 (Memory Proximity Indicator):
在输入框输入时，若系统发现高价值匹配，输入框边框变色或出现闪烁图标，提示用户点击查看建议记忆。
气泡操作栏 (Bubble Action Bar):
[Model Icon]: 点击可弹出模型/渠道/PAPI 变量切换菜单。
[Context Info]: 点击显示该回复参考了哪些 .claude/rules 节点。
[Branching]: 允许基于此气泡分叉出一个新模型、新配置的对话支线。
任务卡片增强 (Plan Mode):
在生成的 Plan 列表中，每一行卡片末尾都有一个模型选择器。
批量修改： 勾选多个 Step，一键将它们全部切换为某个特定的子 AI 或 PAPI 变量。
