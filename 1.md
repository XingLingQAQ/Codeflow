这是一个非常专业且硬核的 GUI 功能规划。根据您的要求，我剔除了所有营销话术，专注于架构设计、配置颗粒度以及状态同步逻辑。

这份规划将 GUI 视为一个多 Agent 编排控制台，而非简单的聊天窗口。

Code Flow V8.0 GUI 功能规划书 (定制开发版)
1. 配置系统：多层级与颗粒度控制 (Configuration & Scope)
核心逻辑： 采用 Global (默认) -> Session (会话级) -> Role (角色级) 的继承覆盖机制。

ID功能模块详细逻辑说明CFG-01全局默认配置 (Global Profile)设定系统的“基准线”。



• 默认的主 AI/Coder/Sub AI 模型。



• 默认挂载的公共 MCP（如：文件系统、Git）。



• 全局 API 渠道池（管理所有 Key 和 BaseURL）。

CFG-02会话级配置 (Session Override)创建新对话或编辑现有对话时，允许完全重写配置。



• 场景： Session A 做 Python 开发（配置 Coder=DeepSeek-Coder）；Session B 写小说（配置 Main=Claude-Opus，禁用 Coder）。



• 存储： 配置随 Session ID 持久化，重启不丢失。

CFG-03角色级精细化配置 (Per-Role)在一个 Session 配置页中，需为三个角色（Main, Sub, Coder）分别独立配置：



1. Model: 具体模型参数 (Temp/TopP)。



2. API Channel: 指定该角色走哪个渠道（如 Coder 走硅基流动，Main 走官方）。



3. System Prompt: 角色的人设指令。



4. Exclusive MCP: (关键) 仅挂载给该角色的工具。例如：Coder 挂载 FileSystem，Sub AI 挂载 GoogleSearch，Main AI 挂载 Orchestrator。

2. AI 模式与协作编排 (AI Modes & Orchestration)
核心逻辑： 基于“工具调用”的控制流，结合 Code Flow 的“对话总结”机制来压缩上下文。

ID功能模块详细逻辑说明MODE-01单 AI 模式 (Standalone)• 传统的 Request-Response 模式。



• 仅加载 Main AI 配置。



• 屏蔽 Sub/Coder 的 UI 元素，降低干扰。

MODE-02多 AI 协作模式 (Orchestration)• 主 AI (Commander): 也就是 Main AI，它是唯一直接与用户对话的接口。



• 工具化调用: Coder 和 Sub AI 不直接回复用户，而是封装为 Function/Tool (如 call_coder_agent, consult_sub_expert) 供 Main AI 调用。



• UI 表现: 当 Main AI 调用工具时，GUI 需在主消息流中渲染一个嵌套的“子对话框”，展示 Coder/Sub 的执行过程和输出。

MODE-03Code Flow 对话总结集成• 自动触发: 当协作轮次过多导致 Context 告警时，触发 Code Flow 的总结功能。



• 手动触发: 在调用 Coder 之前，GUI 允许将当前冗长的用户对话先 Summarize，再将总结后的“任务简报”作为 Prompt 发送给 Coder，节省 Coder 的 Token。

3. 状态管理：Git 与对话的双向回滚 (Bi-directional Rollback)
核心逻辑： “代码文件”与“对话记忆”必须是原子性绑定的 (Atomic Binding)。

ID功能模块详细逻辑说明SYNC-01原子快照 (Atomic Snapshot)每次 Coder AI 准备写入代码前，系统生成一个 Snapshot ID，包含：



1. Git Commit Hash (代码状态)



2. Conversation State JSON (包含 Main, Sub, Coder 三方的完整对话历史栈)



3. Vector Store Index (当前的向量库状态指针)

SYNC-02全栈回滚 (Full-Stack Revert)当用户点击“回滚到此步骤”时：



1. FS 层: 执行 git reset --hard <commit_hash>。



2. Memory 层: 关键步骤，清空当前数据库中该 Snapshot 之后的所有 Message 记录，将 AI 的“短期记忆”强行重置回该时刻。



3. UI 层: 界面上的对话气泡即使已经生成，也必须产生视觉上的“撤回”或“变灰废弃”效果。

SYNC-03子 AI 状态同步如果回滚发生时，Sub AI 正在进行一段独立的思考链，该回滚操作必须发送 Kill Signal 终止子 AI 进程，并丢弃其生成的未完成 Token。

4. GUI 交互设计 (Interaction Design)
为了实现上述功能，建议的 GUI 布局结构：

左侧：会话与配置栏
会话列表 (Session List)
当前会话配置面板 (Active Config Panel): 点击齿轮图标展开，显示三个 Tab：[主 AI] [Coder] [子 AI]。每个 Tab 下配置模型、MCP 开关、API 密钥。
中间：主编排流 (Orchestration Stream)
用户气泡。
Main AI 气泡。
协作卡片 (Collapsible Card): 当 Main AI 调用 Coder 时，显示一个卡片 🔨 Coder AI Working...。
展开卡片可见：Coder 的 Input (由 Main AI 总结的任务) + Coder 的 Output (代码/日志)。
右侧：辅助与状态
CodexLens 视图: 当前引用的文件/Context。
Git 时间轴: 竖向的时间轴，每个节点代表一次 AI 修改。
Action: 鼠标悬停显示 Diff。
Action: 点击“Revert”触发 SYNC-02 逻辑。
实现提示 (Technical Notes)
工具定义 (Tool Definitions):
你需要为主 AI 定义特殊的 Tool Schema，例如：
JSON

{
"name": "delegate_coding_task",
"description": "Delegate specific coding implementation to the Coder AI.",
"parameters": {
"type": "object",
"properties": {
"task_summary": { "type": "string", "description": "Summarized requirements for the coder, excluding chat noise." },
"target_files": { "type": "array", "items": { "type": "string" } }
}
}
}
GUI 后端拦截这个 Tool Call，将 task_summary 转发给 Coder 配置的模型 API。
数据结构 (State Management):
建议使用 SQLite 存储对话，表结构需包含 snapshot_id 字段，以便通过 DELETE FROM messages WHERE snapshot_id > target_id 实现快速回滚。
